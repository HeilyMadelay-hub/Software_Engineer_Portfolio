Razón de los servicios y decisiones arquitectónicas

El proyecto adopta una arquitectura MVC con un pipeline de reconocimiento desacoplado, 
diseñada específicamente para un sistema de visión por computador en tiempo real. La comunicación continua entre frontend y backend se realiza mediante SignalR, 
manteniendo una conexión persistente que permite procesar gestos de forma fluida y sin interrupciones.

La lógica del sistema se divide en servicios especializados, cada uno con una responsabilidad clara, expuestos mediante interfaces para facilitar 
la inyección de dependencias, el desacoplamiento y la mantenibilidad:

HandLandmarkService
Encargado exclusivamente de la extracción de landmarks (84 coordenadas) a partir del modelo de visión. 
Su responsabilidad se limita a la obtención de datos geométricos, sin aplicar lógica de clasificación.

GestureSignatureService
Implementa el algoritmo matemático , comparando los landmarks recibidos con las firmas
almacenadas en archivos JSON mediante similitud coseno y umbrales adaptativos. Este servicio contiene la lógica de reconocimiento propiamente dicha.

GestureDetectorService (Orquestador)
No ejecuta cálculos pesados ni algoritmos de visión. Su función es coordinar
el flujo del pipeline, invocando a los servicios anteriores en el orden correcto y devolviendo un resultado final coherente.
Gracias a este orquestador, el CameraHub permanece limpio y libre de lógica compleja, limitándose a la comunicación en tiempo real.

Una analogía sencilla sería la producción de una película:

🎥 Cámara → graba las escenas (HandLandmarkService)

🎼 Músicos → interpretan la partitura (GestureSignatureService)

🎬 Director → decide qué ocurre y cuándo (GestureDetectorService / Orquestador)

Esta separación de responsabilidades permite un sistema claro, extensible y fácil de razonar, donde cada componente hace una sola cosa y la hace bien.

Por qué no se utilizan DTOs ni API REST

El sistema funciona exclusivamente en tiempo real mediante SignalR, lo que requiere una conexión persistente y continua para la detección de gestos.
No se implementa una API REST ni DTOs independientes porque:

No existen clientes externos ni consumo público de endpoints.

No hay una frontera de sistema tradicional basada en peticiones HTTP.

Los modelos se utilizan únicamente de forma interna dentro del pipeline de procesamiento.

Introducir DTOs o endpoints REST añadiría latencia, serialización innecesaria y rompería la continuidad temporal del flujo de vídeo.

Esta decisión responde directamente a los requisitos de un sistema de visión por computador en tiempo real, donde la baja latencia, 
la simplicidad del flujo y la estabilidad son prioritarias frente a una arquitectura orientada a servicios tradicionales.
